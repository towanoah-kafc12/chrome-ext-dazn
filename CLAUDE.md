# AI駆動開発テンプレート運用ガイド

このリポジトリは、アイデアをAIエージェントに渡してから、要件整理・意思決定・実装・レビューまでを毎回同じ品質で進めるためのテンプレートです。作業するエージェントは、本ファイルと `docs/` 配下の記録を必ず参照し、判断根拠や未解決事項を残してください。

## 基本方針

- 作業開始時に、本ファイル、`MEMO.md`、および `docs/` 配下の関連ファイルを確認する。
- 不明点、意思決定、レビュー指摘は口頭やチャットだけで終わらせず、該当する `docs/` 配下の記録に反映する。
- 実装・修正時は、関連する pending / decision / review の記録を参照し、未解決事項や既存の決定に反しないことを確認する。
- 実装に移る前に、`docs/00_base/` 配下のドキュメントを埋め、目的、要件、実装方針、作業分解、検証方針をある程度固める。
- `docs/00_base/` 配下を埋めるために必要な情報が不足している場合は、ユーザへ質問し、その回答を基に内容を組み立てる。
- ユーザは `docs/00_base/00_idea.md` をアタッチしたプロンプトで現在のアイデアを提示するため、AIエージェントはそれに応じて `docs/00_base/` 配下の内容を埋める作業を主導する。
- 調査、検索、実装、テストなど負荷のある作業ではサブエージェントをフル活用し、メインエージェントはオーケストレーション、指示、管理、判断を中心に実行する。
- 複数のAIエージェントツールで作業する可能性があるため、Codex は `AGENTS.md`、Claude Code は `CLAUDE.md` を入口として同等の指示を参照できる状態に保つ。
- `AGENTS.md` を更新した場合は、原則として同じ内容を `CLAUDE.md` にも反映する。

## docs ディレクトリの役割

`docs/` 配下には、AI駆動開発中に発生する未解決事項、決定事項、レビュー指摘を蓄積します。各フォルダ内に作成する Markdown ファイル名は、機能や作業単位が分かる名称を先頭に付け、末尾に用途を付けてください。

例:

- `docs/00_base/00_idea.md`
- `docs/00_base/01_implementation.md`
- `docs/01_pending/auth_pending.md`
- `docs/02_decision/auth_decision.md`
- `docs/03_review/auth_review.md`

### docs/00_base/00_idea.md

アイデアだけの状態から、目的、想定ユーザ、MVP、非目標、成功条件を整理し、実装に進める粒度まで詳細化するためのテンプレートです。

### docs/00_base/01_implementation.md

使う技術、設計方針、実装方針、実装スコープ、テスト方針をある程度確定させ、実装時に参照できる状態で保存するためのテンプレートです。

### docs/00_base/02_task_breakdown.md

実装方針を、AIエージェントが安全に実行できる作業単位へ分解するためのテンプレートです。作業順序、変更対象ファイル、完了条件、確認方法、依存関係を明確にします。

### docs/00_base/03_verification.md

実装後に、目的、MVP、タスク完了条件、テスト方針を満たしているか確認するためのテンプレートです。自動テスト、手動確認、未確認事項、残課題、完了判断を記録します。

### docs/01_pending

不明点、情報不足、ユーザ確認が必要な事項、調査待ちの事項を記録します。

必須項目:

- 項番
- 日付
- タイトル
- 対象ファイル
- 内容
- 対応方針
- 状態（未解決 / 解決済）

記載形式:

1. ファイル上部に Markdown の表を置き、項番、日付、タイトル、対象ファイル、状態を一覧化する。
2. ファイル下部に `## <項番>: <タイトル>` のセクションを作り、内容、対応方針、補足情報を Markdown 形式で詳述する。
3. 対応方針が未定の場合は空欄でもよいが、状態は必ず明記する。

### docs/02_decision

エージェントとユーザの対話で決定したこと、新たに判明した重要情報、判断理由、設計上の前提を記録します。

必須項目:

- 項番
- 日付
- タイトル
- 内容

記載形式:

1. `## <日付>: <タイトル>` のセクション単位で記載する。
2. セクション内に項番、内容、判断理由を含める。
3. 対応が必要なファイル、関連情報、代替案、影響範囲がある場合は、必要に応じて小セクションを追加する。

### docs/03_review

エージェントのレビュー結果、ユーザからの追加指摘、未対応の改善指摘を記録します。

必須項目:

- 項番
- 日付
- 対象ファイル
- 指摘内容
- 対応方針
- 状態（- / 修正済）

記載形式:

1. ファイル上部に Markdown の表を置き、項番、日付、対象ファイル、指摘内容、対応方針、状態を一覧化する。
2. 指摘内容が複雑または長くなる場合は、ファイル下部に `## <項番>: <タイトル>` のセクションを作り、詳細を記載する。
3. 修正が完了したら状態を `修正済` に更新し、必要に応じて関連する decision も更新する。

## 作業時の確認事項

作業時には以下の事項を確認しつつ作業することを厳守する。

- 作業開始前に `AGENTS.md` / `CLAUDE.md` / `MEMO.md` を確認する。
- 関連する `docs/00_base/00_idea.md` / `docs/00_base/01_implementation.md` の前提と方針を確認する。
- 関連する `docs/00_base/02_task_breakdown.md` の作業順序、変更対象、完了条件を確認する。
- 関連する `docs/00_base/03_verification.md` の検証方針、受け入れ条件、未確認事項を確認する。
- 関連する `docs/01_pending` の未解決事項を確認する。
- 関連する `docs/02_decision` の決定事項を確認する。
- 関連する `docs/03_review` の未対応指摘を確認する。
- 新しい不明点、決定、レビュー指摘が発生した場合、該当するファイルに記録する。
- `AGENTS.md` を変更した場合、`CLAUDE.md` にも同じ内容を反映する。
